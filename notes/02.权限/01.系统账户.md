## 1. bin
- 传统用途：在早期的Unix系统中，`bin`用户拥有`/bin`目录下的可执行命令文件。这是一种历史遗留的权限管理模式。
- 现在状态：基本不再使用，`/bin`目录的权限通常设置为`root:root`
- 存在意义：为向后兼容而保留
```bash
$ grep ^bin /etc/passwd
bin:x:1:1:bin:/bin:/sbin/nologin # UID 1，非常古老
```

## 2. daemon
- 核心用途：用于运行**系统守护进程**(deamon)
- 运行场景：某些不需要与用户交互、只需要在后台运行的系统级服务，可能会选择以 `daemon` 身份运行，以获取比`root`更低的权限。
- 与`nobody`的区别：权限通常略高于`nobody`，用于运行那些需要少许系统访问权限的后台服务（例如，一些老版本的打印服务 `lpd`)
- 现代实践：现在也不常用。更佳实践是为每个独立的守护进程创建专门的系统用户(如： `nginx`、`mysql`、`redis`等)。
- 权限设计：
```bash
daemon:x:2:2:daemon:/sbin:/sbin/nologin
```
## 3. adm
- 核心用途：与系统监控和日志文件的读取权限相关。
- 关键目录：`/var/log` （系统日志目录）
- 工作原理：许多系统日志文件（如 `/var/log/syslog`、`/var/log/auth.log`)的组权限被设置为`adm`。这样，属于`adm`组的用户/进程就可以读取日志，而无需拥有`root`权限。
- 使用场景：系统监控工具、日志分析工具可以以`adm`组成员的身份运行，安全地访问日志。
- 权限设计：
```bash
$ ls -ld /var/log
drwxr-xr-x 1 root adm /var/log  # 目录属于 root，但组是 adm
drwxr-xr-x 1 root root 85 9月  15 20:20 /var/log # 我电脑上输出是这个，组不是adm
```

## 4. nobody
- 核心用途：最低权限通用账户，主要用于运行不需要任何特权的网络服务。
- 权限等级：最低，几乎没有任何文件的所有权，属于 `nogroup`。
- 现代使用：仍然非常常用，是运行非特权服务的标准选择。

## 总结对比表

|账户名|典型 UID|主要用途|现代使用频率|权限等级|
|---|---|---|---|---|
|**bin**|1|管理 `/bin` 命令 (历史遗留)|极低，已废弃|低 (但比 nobody 高)|
|**daemon**|2|运行系统守护进程|低，被专用账户取代|低到中|
|**adm**|3 或 4|读取系统日志 (`/var/log`)|**中等**，仍有特定用途|低 (但有日志读取权)|
|**nobody**|99, 65534|运行无特权网络服务|**非常高**，标准实践|**最低**|

## 核心设计思想
这些账户的存在，共同体现了Linux/Unix系统的两个根本安全原则：
1. **最小权限原则**：进程只拥有完成其任务所必需的最小权限。一个邮件服务进程不需要能修改系统日志，一个网页服务器进程也不需要能关机。
2. **职责分离**：不同的任务由不同的、隔离的账户执行。文件管理(`bin`)、日志处理(`adm`)、后台服务(`daemon`)、匿名访问(`nobody`)各司其职。

## 给系统管理员的建议
1. **不要删除或随意修改这些账户**：它们是系统基础设施的一部分。
2. **理解并利用 `adm` 组**：如果您编写了一个需要读取系统日志的监控脚本，可以考虑将其运行用户加入 `adm` 组，而不是赋予 `root` 权限。
3. **效仿这种模式**：在部署自己的应用时，**为每个服务创建专用的系统用户**（例如 `myapp`），并遵循最小权限原则。不要使用 `nobody` 运行所有服务，因为这会破坏服务间的隔离性。
4. **检查现有服务**：使用 `ps aux` 或 `systemctl status <服务名>` 查看您的服务正以哪个用户身份运行。
